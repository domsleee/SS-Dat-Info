# Decompiled Source Code - AI Assistant Guide

## Overview
This directory contains decompiled source code from Supreme Snowboarding (1999), Housemarque's 3D Engine. These are the **authoritative source files** used to verify and document the HX file format.

**See also**: `../HX_FILE_FORMAT.md` for the documented file format specification. This guide explains **how to USE the decompiled files** to verify, extend, and debug that documentation.

## File Extensions & Formats

### `.c` Files (Decompiled C Code)
- **Purpose**: High-level pseudocode representation of the original game engine
- **Tool**: Generated by Ghidra decompiler
- **Language**: C-like pseudocode (not compilable, but human-readable)
- **Use for**: Understanding algorithms, data structures, and business logic
- **Examples**: `sr.c`, `HMG_3DE.c`

### `.asm` Files (Annotated x86 Assembly)
- **Purpose**: Low-level assembly language from the original executable
- **Architecture**: x86 (32-bit)
- **Tool**: Disassembled by Ghidra with annotations
- **Use for**: 
  - Verifying exact byte offsets in structures
  - Understanding memory layout details
  - When C decompilation is ambiguous
  - Finding exact numerical constants
- **Examples**: `sr.asm`, `HMG_3DE.asm`

**Note**: The `.asm` extension is appropriate for x86 assembly. No change needed.

## Key Files

### `sr.c` - SurRender Engine Core
- **Primary UV Generation**: `srModeler::planarMap()` at **line 64537** (address `0x43e70`)
- **Contains**:
  - Planar UV projection algorithm
  - Bounding box calculations (`getAxialBounds`)
  - MappingInfo structure handling
  - Vertex UV assignment logic

### `HMG_3DE.c` - 3D Engine & File Loading
- **Primary Mesh Loading**: Chunk-based file reader starting around line 3300
- **Contains**:
  - HX file format chunk reading
  - Chunk type handlers (0x01 triangles, 0x02 vertices, etc.)
  - Memory allocation and structure building
  - Vertex/triangle data parsing

### `sr.asm` - Assembly Reference
- **Use when**: You need exact byte offsets or low-level memory operations
- **Cross-reference with**: `sr.c` for understanding context

### `HMG_3DE.asm` - Assembly Reference
- **Use when**: Verifying chunk sizes, structure layouts
- **Cross-reference with**: `HMG_3DE.c` for understanding file format details

## How to Use These Files as an AI Assistant

### 1. Verifying File Format Documentation
When asked to verify `HX_FILE_FORMAT.md`:

```
1. Search HMG_3DE.c for chunk reading functions
2. Look for patterns like: case 0x01, case 0x02, etc.
3. Find memory reads: getInt32(), getFloat32()
4. Count bytes read per chunk entry
5. Update HX_FILE_FORMAT.md if discrepancies found
```

**Example**: To verify chunk 0x02 (vertices):
- Search: "case 2:" or "0x02" in `HMG_3DE.c`
- Find: Three `getFloat32()` calls = 12 bytes per vertex
- Verify: Matches `HX_FILE_FORMAT.md` "3 × float32 (X, Y, Z)"
- If mismatch: Update documentation with source code evidence

### 2. Finding Algorithm Implementations
When asked "How does X work?" or "Where is Y implemented?":

```
1. Search sr.c for function names (e.g., "planarMap", "getAxialBounds")
2. Read the function implementation
3. Extract formulas and logic
4. Add to HX_FILE_FORMAT.md with line number citations
```

**Example**: UV generation was NOT documented until we found `planarMap()` in sr.c line 64537.

### 3. Understanding Data Structures
When investigating memory layouts or structure fields:

```
1. Find structure usage in .c file (e.g., "pMVar3 + 8")
2. Check .asm file for exact offset verification
3. Note: Ghidra uses pointer arithmetic (pMVar3 + 8 = offset +0x08)
4. Document structure layout with field names and offsets
```

**Key insight**: The decompiled code shows HOW data is accessed, not necessarily how it's stored in files.

### 4. Coordinate System & Transformations
**CRITICAL**: The game uses a **LEFT-HANDED Y-DOWN** coordinate system:
- X+ = Right
- Y+ = DOWN (not up!)
- Z+ = Into screen

When implementing for modern engines (Three.js uses Y-up):
- **You MUST transform coordinates** when Y-down → Y-up
- Check projection plane selection (XY in Y-down ≠ XY in Y-up)
- See `sr.c` for axis handling in planarMap function

**This is the #1 cause of "texture is mirrored/flipped" bugs!**

### 5. Working with Formulas
Translating Ghidra's auto-generated variable names to readable formulas:

**Example from sr.c line 64567-64587**:
```c
local_c = *(float *)(pMVar3 + 8) / (local_8 - local_10);
// Translates to: scale_factor_u = mapping.scale_u / (bbox_max_u - bbox_min_u)

pfVar5[0] = (fVar3 - local_10) * local_c + fVar4;
// Translates to: U = (vertex_u - bbox_min_u) * scale_factor_u + offset_u

pfVar5[1] = (-fVar1 - (float)param_3) * local_14 + fVar2;
// Translates to: V = (-vertex_v - bbox_min_v) * scale_factor_v + offset_v
```

**Note the V-axis negation** (`-fVar1`) - this is intentional and part of the algorithm!

**Document these formulas** in `HX_FILE_FORMAT.md` UV Generation section with line citations.

### 6. Debugging Implementation Issues
When user reports rendering problems:

```
1. Check coordinate system transformation (Y-down vs Y-up)
2. Verify projection plane (XY vs XZ vs YZ)
3. Check for V-axis negation in UV formula
4. Look for face normal handling in sr.c
5. Consider separate texture regions for front/back faces
```

### 7. Variable Naming Convention in Ghidra
Ghidra auto-generates variable names:
- `local_XX`: Local variables (stack)
- `param_X`: Function parameters
- `pMVar3`, `fVar4`: Pointers and floats (type prefix)
- `iVar6`: Integer variable
- Numbers after underscore are stack offsets

**Translate to meaningful names** when documenting in `HX_FILE_FORMAT.md`!

```c
offset +0x00: axis1 (int)
offset +0x04: axis2 (int)  
offset +0x08: scale_u (float)
offset +0x0C: scale_v (float)
```

- **Stride**: 872 bytes (`0x368`)
- **UV offset**: `(stage + 0x1e + layer * 2) * 8`

## Key Differences: File Format vs Engine Runtime

| Aspect | HX File (on disk) | Engine Runtime (in memory) |
|--------|-------------------|---------------------------|
| **Vertices** | Stored (chunk 0x02) | Loaded + transformed |
| **Triangles** | Stored (chunk 0x01) | Indexed references |
| **UVs** | **NOT STORED** | Generated via planarMap |
| **Normals** | Stored (chunk 0x05) | Loaded as-is |
| **Vertex structure** | 12 bytes (X,Y,Z) | 872 bytes (full vertex data) |

**Lesson**: Decompiled code shows BOTH file reading AND runtime processing. Distinguish between them!
